Agreed:

The team agreed that the core innovation of the project is an agent-based HVAC temperature control system that turns natural-language or voice-style user commands into control actions. The project will focus on the agent’s role in understanding user intent and making control decisions, instead of treating PID itself as the main contribution.
The team agreed to define the system as a closed-loop architecture. The agent is not only used once at the beginning to interpret the user request, but also works as an online supervisor that checks controller performance and updates decisions during operation.
The team agreed that the agent should make decisions based on both user intent and contextual information. The current design includes information such as weather, electricity tariff, user schedule, room state, user habits, solar irradiance, and PID telemetry, with data stored in SQLite.
The team agreed to use a 2R2C thermal-network digital twin as the main experiment environment, with both a residential room setting and a larger commercial-space setting, so the project can test whether the method works across different thermal conditions.
The team agreed on a four-condition baseline design. The comparison includes a fixed PID baseline without LLM, an LLM setpoint-only version with the same PID settings, an LLM-in-the-loop reactive version, and an LLM-in-the-loop proactive version with forecast information. This setup is intended to separate the value of simple LLM assistance from the value of full agent supervision.
The team agreed on six main test scenarios, covering steady tracking, weather disturbance, cost-sensitive control, occupancy change, multi-disturbance stress, and LLM failure resilience. These scenarios were chosen to test comfort, cost, robustness, and decision quality under different operating conditions.
The team agreed on an evaluation framework with four dimensions: thermal comfort, energy efficiency, responsiveness, and stability. The metrics include temperature deviation, energy and tariff-related cost, settling and response behavior, and oscillation or overshoot after adjustment.

Learned:

The team learned that the agent’s role should be described more concretely. It is not only parsing user language, but also producing structured control outputs such as target temperature range, HVAC mode, fan preference, and decision rationale, while using tools to enrich its decision.
The team learned that the control logic can be framed as multi-objective rather than single-objective. Instead of only tracking temperature, the design now explicitly considers comfort, energy use, and response behavior together, which better matches the kinds of user requests the project wants to support.
The team learned that proactive control should be part of the story. The current method is not limited to reacting after temperature changes happen, but also includes ideas such as adding compensation terms and adjusting the effective setpoint ahead of time to handle predicted disturbances and thermal delay.
The team learned that robustness needs to be addressed explicitly because small local LLMs may fail in tool-calling or output formatting. The current design already includes fallback strategies, validation rules, safe bounds, and missing-field correction, which strengthens the practicality of the project.
The team learned that the experiment design is now specific enough to support meaningful comparison. The conditions are arranged progressively, so each added capability can be tested more fairly, and the scenarios connect directly to realistic user requests such as keeping the room comfortable, reducing cost, or preparing for occupancy changes.
The team learned that the project already has a solid evaluation and logging plan. The system records thermal states, control signals, cost-related values, and LLM decisions, which means later analysis can compare not only final performance but also how and why the controller behaved differently across settings.
