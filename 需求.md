# Home Temperature Control Agent

## 1. 项目简介

本项目实现一个面向 Home Assistant 的温控 Agent，用于解析用户自然语言指令，并将其转换为结构化的高层控制配置，再交由 PID 控制器进行自动闭环控制。

系统职责划分如下：

- **Agent**
  - 解析用户自然语言
  - 提取房间、目标温度、模式、约束条件
  - 输出结构化配置

- **PID Controller**
  - 读取当前温度
  - 根据目标温度进行自动控制
  - 输出控制量

- **Home Assistant**
  - 提供设备状态
  - 执行 climate API/service 调用

---

## 2. 系统目标

支持以下能力：

1. 用户通过自然语言描述温控需求  
2. Agent 将自然语言转换为控制目标  
3. Agent 预测环境变化趋势，调节 PID 的配置
4. PID 根据目标温度自动控制空调/暖气  
5. Home Assistant 执行具体设备动作  

示例用户输入：

- “客厅调凉一点”
- “今晚卧室保持 24 度，省电优先”
- “没人时把温度调高一点”
- “睡觉时不要风太大”

示例 PID 配置调整 based on 环境变化趋势

- 预测温度即将快速升高，原本应该提供给PID控制系统是 8 度室内外温差，改成 10 度，以应对快速升温
- 通过预测算法，对 Kp，Ki，Kd 进行微调
- 面对快速升温，调高提供给 PID 系统的输出功率，20 W --> 25 W

## 3. 系统架构

```text
用户自然语言
    ↓
Agent（自然语言理解）
    ↓
结构化目标配置
    ↓
PID Controller（自动控制）
    ↓
Home Assistant API
    ↓
空调 / 暖气 / 风扇
    ↓
温度传感器反馈
    ↺
```

## 4. Agent 的职责

Agent 不负责直接进行闭环控制，也不负责实时调整 Kp / Ki / Kd。

Agent 只负责输出**高层控制配置**，例如：

* `room`
* `target_temperature`
* `hvac_mode`
* `preset_mode`
* `fan_mode`
* `deadband`
* `valid_until`

---

## 5. PID 的职责

PID 控制器负责：

* 获取当前温度
* 读取目标温度
* 计算误差
* 根据 `Kp`, `Ki`, `Kd` 输出控制量
* 将控制量映射为 Home Assistant API 动作

PID 不负责解析自然语言。

## 8. Agent 输出约束

Agent 必须遵守以下规则：

1. 只输出合法 JSON
2. 不直接输出 Home Assistant API 调用
3. 不直接输出 PID 参数 (`Kp`, `Ki`, `Kd`)
4. 温度必须在安全范围内，例如 `[20, 28]`
5. 当用户描述模糊时，优先生成保守配置
6. 当窗户打开时，可建议暂停制冷
7. 当用户要求与当前上下文冲突时，输出最安全方案

## 8.5 Agent Tool Use

Agent 支持通过 tool use（函数调用）主动获取外部信息，以做出更精准的温控决策。Agent 在推理过程中可按需调用以下 tools，获取返回结果后综合判断，再输出最终 JSON 配置。

### 调用流程

```text
用户指令
    ↓
Agent（LLM with tool use）
    ├── 可选：调用 get_weather()
    ├── 可选：调用 get_user_habits()
    ├── 可选：调用 get_room_status()
    ├── 可选：调用 get_energy_price()
    ├── 可选：调用 get_schedule()
    ↓
Agent 综合用户指令 + tool 返回数据，输出 JSON 配置
    ↓
PID Controller
```

### 可用 Tools

#### get_weather

查询指定城市的当前室外天气信息。

* **参数**：`city` (string) — 城市名称，如 `"Pittsburgh"`
* **返回值**：

```json
{
  "outdoor_temp": 5.2,
  "humidity": 72,
  "condition": "rain",
  "wind_speed_kmh": 15
}
```

* **用途**：室外温度影响室内散热速率。室外低温时不需要过度制冷；室外高温高湿时需要更积极制冷。

#### get_user_habits

查询用户的历史温控偏好。

* **参数**：`user_id` (string)，`time_of_day` (string，如 `"night"` / `"morning"`)
* **返回值**：

```json
{
  "preferred_sleep_temp": 23.0,
  "preferred_day_temp": 25.0,
  "bedtime": "23:00",
  "wake_time": "07:30",
  "fan_preference": "low"
}
```

* **用途**：根据用户历史偏好做出更个性化的温度决策，减少"猜测"。

#### get_room_status

查询房间的实时传感器状态（替代 `current_context` 的手动组装）。

* **参数**：`room` (string) — 房间 ID
* **返回值**：

```json
{
  "current_temperature": 27.2,
  "current_humidity": 65,
  "current_hvac_mode": "cool",
  "current_fan_mode": "medium",
  "occupied": true,
  "window_open": false
}
```

* **用途**：Agent 可自主查询房间状态，而非依赖调用方预组装上下文。

#### get_energy_price

查询当前电价信息（峰谷电价）。

* **参数**：`time` (string，ISO 8601)
* **返回值**：

```json
{
  "price_per_kwh": 0.85,
  "tier": "peak",
  "next_off_peak": "2026-03-30T00:00:00"
}
```

* **用途**：峰时电价时 Agent 应倾向 `preset_mode: eco`；谷时电价时可适度提升舒适度。

#### get_schedule

查询用户的日程安排。

* **参数**：`user_id` (string)，`date` (string，如 `"2026-03-29"`)
* **返回值**：

```json
{
  "events": [
    {"type": "leave_home", "at": "08:30"},
    {"type": "return_home", "at": "18:00"}
  ]
}
```

* **用途**：用户即将外出时可提前切换 eco 模式；即将回家时提前预冷/预热。

### Tool Use 规则

1. Tool 调用是**可选**的，Agent 可根据需要决定是否调用
2. 若 tool 调用失败（超时、服务不可用），Agent 应基于已有的 `current_context` 继续推理，不得因 tool 失败而拒绝输出
3. Agent 每次推理最多调用 **3 次** tool，避免延迟过高
4. Tool 返回的数据仅作为辅助参考，最终决策权仍遵循 §8 的输出约束
5. Agent 在 `reason` 字段中应注明参考了哪些 tool 数据

## 10. PID 控制逻辑

PID 控制器读取以下输入：

* `current_temperature`
* `target_temperature`
* `dt`
* `prev_error`
* `integral`
* `Kp`, `Ki`, `Kd`

### 误差定义

```text
error = target_temperature - current_temperature
```

### PID 公式

```text
u = Kp * error + Ki * integral + Kd * derivative
```

其中：

```text
integral = integral + error * dt
derivative = (error - prev_error) / dt
```

---

## 11. 控制输出映射

PID 输出的是控制量 `u`，需要映射成 Home Assistant 的动作。

### 示例映射规则

```text
u < -2.0      → 强制制冷，设定温度更低
-2.0 <= u < -0.5 → 普通制冷
-0.5 <= u <= 0.5 → 保持当前状态
u > 0.5       → 停止制冷或切换加热
```
